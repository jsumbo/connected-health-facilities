# Deploy: Render (API) + Vercel (frontend)

Split deployment for the Connected Facilities dashboard:

| Layer | Platform | Directory |
|-------|----------|-----------|
| FastAPI API | [Render](https://render.com) | `backend/` |
| Next.js UI | [Vercel](https://vercel.com) | `frontend/` |

---

## 1. Render — backend API

### Create the service

1. Push this repo to GitHub/GitLab.
2. Render → **New** → **Blueprint** → connect the repo → apply `render.yaml`.
3. Or **New Web Service** → Docker → set **Root Directory** to `backend`.

### Required secrets (Render Dashboard → Environment)

| Variable | Example / notes |
|----------|-----------------|
| `KOBO_API_TOKEN` | KoboToolbox API token |
| `KOBO_ASSET_UID` | Baseline assessment form asset UID |
| `KOBO_SENTIMENT_ASSET_UID` | Staff sentiment form asset UID |
| `DASHBOARD_USERNAME` | Admin login username |
| `DASHBOARD_PASSWORD` | Strong password |
| `CORS_ALLOWED_ORIGINS` | Vercel URLs (comma-separated, no trailing slash) |

`JWT_SECRET_KEY` is auto-generated by the Blueprint.

### After deploy

Note the public URL, e.g. `https://datflow-api.onrender.com`.

Verify:

```bash
curl https://YOUR-API.onrender.com/health
```

Expect `"status": "ok"` and cache fields populated after the first Kobo refresh (~30–60s on cold start).

### DLA CSV

The Blueprint sets `DLA_CSV_PATH` to the file baked into the Docker image under `backend/data/`. To update the CSV, commit a new file and redeploy.

### CORS

Set `CORS_ALLOWED_ORIGINS` to every frontend origin that calls the API from the browser:

```
https://your-project.vercel.app,https://readiness.your-domain.gov.lr
```

Include **preview** origins if you test admin login on PR deployments, e.g. `https://your-project-git-branch-team.vercel.app`.

Public pages mostly fetch on the **server** (no browser CORS). **Admin login** posts to `/auth/token` from the browser, so CORS must allow your Vercel origin in production.

## 2. Vercel — frontend

### Create the project

1. [vercel.com/new](https://vercel.com/new) → Import the Git repo.
2. **Root Directory**: `frontend` (required).
3. Framework: **Next.js** (auto-detected).
4. Deploy.

### Environment variables (Vercel → Settings → Environment Variables)

Set for **Production** and **Preview**:

| Variable | Value |
|----------|--------|
| `NEXT_PUBLIC_API_URL` | `https://YOUR-API.onrender.com` (HTTPS, no trailing slash) |
| `NEXT_PUBLIC_SITE_URL` | `https://YOUR-VERCEL-DOMAIN.vercel.app` or custom domain |

`NEXT_PUBLIC_*` values are embedded at **build time**. Redeploy after changing them.

### Custom domain (optional)

Vercel → Project → **Domains** → add e.g. `readiness.health.gov.lr`.

Then update:

- `NEXT_PUBLIC_SITE_URL` on Vercel
- `CORS_ALLOWED_ORIGINS` on Render (add the custom domain)

### Production deploy

```bash
cd frontend
npx vercel link
npx vercel --prod
```

Or push to the production branch with Git integration.

## 3. Wiring checklist

```
Browser → Vercel (Next.js)
              ├─ Server Components → Render /public/* (server-side fetch)
              ├─ Login form → Render /auth/token (browser; needs CORS)
              └─ Cookie → /api/auth/login (same origin on Vercel)

Legacy /dashboard → Render /dashboard/* with Bearer token (browser; needs CORS)
```

- [ ] Render `/health` returns OK
- [ ] `NEXT_PUBLIC_API_URL` points to Render HTTPS URL
- [ ] `NEXT_PUBLIC_SITE_URL` matches the Vercel (or custom) URL
- [ ] `CORS_ALLOWED_ORIGINS` lists all Vercel origins you use
- [ ] Redeploy Vercel after env changes
- [ ] Test `/`, `/facilities`, `/map` on Vercel
- [ ] Test `/login` → `/dashboard` if you use admin routes

## 4. Local development (unchanged)

```bash

cd backend && uvicorn main:app --reload --port 8000

# Terminal 2
cd frontend && NEXT_PUBLIC_API_URL=http://localhost:8000 npm run dev
```

## 5. Troubleshooting

| Symptom | Fix |
|---------|-----|
| Public pages show “Failed to load dashboard data” | Wrong `NEXT_PUBLIC_API_URL` or API down; check Render logs |
| Login fails with network/CORS error | Add Vercel URL to `CORS_ALLOWED_ORIGINS` on Render |
| Stale data after Kobo submit | Wait ~3 min (scheduler) or restart Render service |
| OG previews wrong link | Set `NEXT_PUBLIC_SITE_URL` and redeploy Vercel |
| Render build fails on PORT | Dockerfile uses `$PORT`; redeploy with latest `backend/Dockerfile` |

## 6. Cost notes

- **Render** `starter` web service stays awake for the APScheduler jobs (Kobo/sentiment/DLA refresh every 3 minutes).
- **Vercel** Hobby/Pro is typical for the static/SSR Next app; no long-running jobs on Vercel.
